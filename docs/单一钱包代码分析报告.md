## 系统架构概览

### 整体架构

单一钱包系统采用 **Redis Lua 原子脚本 + 数据库持久化** 的双层架构：

```
三方钱包平台
    ↓
GameController (控制器层)
    ↓
RedisLuaScripts (Redis原子操作)
    ↓
WalletService (钱包服务)
    ↓
PlayerPlatformCash (数据库持久化)
```

### 核心文件清单

| 文件路径 | 职责 | 关键功能 |
|---------|------|---------|
| `app/service/game/SingleWalletServiceInterface.php` | 单一钱包接口标准 | 定义统一的钱包操作接口 |
| `app/service/RedisLuaScripts.php` | Redis Lua 原子脚本 | 提供原子性的余额操作 |
| `app/service/WalletService.php` | 钱包服务 | 带缓存的余额查询和更新 |
| `app/Constants/TransactionType.php` | 交易类型常量 | 统一的交易类型定义 |
| `app/wallet/controller/game/*GameController.php` | 各平台控制器 | 处理三方平台回调 |
| `app/service/game/*ServiceInterface.php` | 各平台服务实现 | 平台特定的业务逻辑 |
| `app/wallet/controller/game/TelegramAlertTrait.php` | 告警通知 | Telegram 异常通知 |

### 1. SingleWalletServiceInterface（单一钱包接口）

**标准接口方法**:

```php
interface SingleWalletServiceInterface {
```

- **已废弃方法**: `bet/cancelBet/betResulet/reBetResulet` 已迁移到 Lua 脚本，但接口未更新
- **命名不一致**: 方法名应统一为驼峰命名

### 2. RedisLuaScripts（Redis 原子操作核心）

**定义文件**: `app/service/RedisLuaScripts.php`

#### 2.1 原子下注 (`atomicBet`)

**功能**: 一次性完成余额检查、扣款、记录保存

**Lua 脚本逻辑**:
```lua
1. 幂等性检查（order:bet:lock:{order_no}）
2. 余额检查（wallet:balance:{player_id}）
3. 扣款（SETEX 更新余额）
4. 设置幂等锁（300秒）
5. 保存下注记录（Hash）
6. 加入同步队列（Sorted Set）
7. 统计计数（INCR）
```

**参数要求**:
- `order_no`: 订单号（必需）
- `amount`: 下注金额（必需，≥0）
- `platform_id`: 平台ID（必需）
- `transaction_type`: 交易类型（必需）
- `original_data`: 原始数据（可选）

**返回值**:
- 成功: `{ok: 1, balance: xxx, old_balance: xxx}`
- 余额不足: `{ok: 0, error: "insufficient_balance", balance: xxx}`
- 重复订单: `{ok: 0, error: "duplicate_order", balance: xxx}`

**⚠️ 问题标注**:
- **幂等锁时间固定**: 300秒幂等锁可能不适合所有场景
- **余额 TTL 短**: 1小时（3600秒）可能导致频繁查询数据库

#### 2.2 原子结算 (`atomicSettle`)

**功能**: 派彩加款 + 更新记录

**Lua 脚本逻辑**:
```lua
1. 幂等性检查（order:settle:lock:{order_no}）
2. 获取当前余额
3. 加款（如果 winAmount > 0）
4. 设置幂等锁（300秒）
5. 检查 bet 记录是否存在
   - 存在：更新 bet 记录
   - 不存在：创建独立 settle 记录
6. 更新同步队列
7. 统计计数
```

**特殊处理**:
- **独立结算**: 如果下注记录不存在，会创建独立的 settle 记录（订单号追加 `_settle`）
- **diff 计算**: `diff = win - bet`（输赢金额）

**⚠️ 问题标注**:
- **订单号后缀处理**: 自动追加 `_settle` 可能导致订单号冲突（如 KT 的立即结算场景）
- **bet 不存在场景**: 缺少对 bet 记录缺失的监控和告警

#### 2.3 原子取消 (`atomicCancel`)

**功能**: 退款 + 更新记录

**Lua 脚本逻辑**:
```lua
1. 幂等性检查
2. 获取当前余额
3. 退款（加钱）
4. 设置幂等锁
5. 更新 bet 记录（如果存在）
6. 更新同步队列
7. 统计计数
```

**⚠️ 问题标注**:
- **无退款金额上限**: 未验证退款金额是否超过原下注金额

### 3. WalletService（钱包服务）

**定义文件**: `app/service/WalletService.php`

#### 3.1 核心功能

| 方法 | 功能 | 缓存策略 |
|------|------|---------|
| `getBalance()` | 查询余额 | 先查缓存，未命中查数据库 |
| `deduct()` | 扣款（数据库） | 使用悲观锁（FOR UPDATE） |
| `add()` | 加款（数据库） | 使用悲观锁 |
| `clearCache()` | 清除缓存 | Redis DEL |
| `updateCache()` | 更新缓存 | Redis SETEX |

#### 3.2 缓存设计

```php
// 缓存键格式（与 Lua 统一）
wallet:balance:{player_id}

// 缓存过期时间
CACHE_TTL = 3600       // 1小时
CACHE_TTL_SHORT = 300  // 5分钟（高频玩家）
```

**🚨 紧急开关**:
```php
// .env 配置
WALLET_CACHE_ENABLED=false  // 禁用缓存
```

**⚠️ 问题标注**:
- **缓存一致性风险**: Lua 脚本直接更新 Redis，WalletService 数据库更新后需手动更新缓存
- **未实现缓存预热**: 高并发场景下可能出现缓存击穿
- **悲观锁性能**: `deduct/add` 使用 FOR UPDATE，高并发下可能成为瓶颈

- `BET_BONUS`: 奖金回合（不扣款）

**结算类（派彩）**:
- `SETTLE`: 普通结算

**取消类（退款）**:
- `CANCEL`: 取消下注
- `CANCEL_REFUND`: 退款
- `CANCEL_ROLLBACK`: 交易回滚

**⚠️ 问题标注**:
- **未强制使用**: 代码中混用字符串和常量
- **缺少类型验证**: 没有运行时验证传入的类型是否合法

## 业务流程详解

### 流程1: 下注流程（以 KT 平台为例）

```
1. KTGameController::bet() 接收请求
   ↓
2. 验证签名（verifyToken）
   ↓
3. 获取玩家信息（Player::query()->where('uuid', $username)）
   ↓
4. 构建 Lua 参数
   - order_no: 订单号
   - amount: 下注金额
   - platform_id: 平台ID
   - transaction_type: TransactionType::BET
   ↓
5. 参数验证（validateLuaScriptParams）
   ↓
6. 执行 Lua 原子下注（RedisLuaScripts::atomicBet）
   ↓
7. 审计日志（logLuaScriptCall）
   ↓
8. 处理返回结果
   - 余额不足: 返回 API_CODE_AMOUNT_OVER_BALANCE
   - 重复订单: 返回成功 + 当前余额
   - 成功: 返回新余额
   ↓
9. 如果 TakeWin=1，立即结算
   - 获取下注记录（Redis）
   - 计算 diff = win - bet
   - 执行 Lua 原子结算（atomicSettle）
   - 使用独立订单号（{order_no}_settle）
```

**⚠️ 问题标注**:
- **立即结算订单号冲突**: KT 的 TakeWin=1 场景，手动追加 `_settle`，但 Lua 脚本也会追加，导致重复
- **bet 记录查询失败**: 如果 Redis 中 bet 记录不存在，diff 计算错误（betAmount=0）

---

### 流程2: 延迟结算流程（以 KT 平台为例）

```
1. KTGameController::betResult() 接收请求
   ↓
2. 解密数据（decrypt）
   ↓
3. 从 Redis 获取下注金额
   - 键: game:record:bet:KT:{order_no}
   - 取值: amount
   ↓
4. 计算 diff = win - bet
   ↓
5. 执行 Lua 原子结算（atomicSettle）
   ↓
6. 处理重复结算（幂等性）
   ↓
7. 返回新余额
```

**⚠️ 问题标注**:
- **依赖 Redis**: 如果 bet 记录已过期（7天 TTL），diff 计算错误
- **缺少降级方案**: Redis 不可用时无法计算 diff

---

### 流程3: 取消/退款流程

```
1. GameController::cancelBet/refund() 接收请求
   ↓
2. 验证签名/解密
   ↓
3. 构建 Lua 参数
   - order_no: 订单号
   - refund_amount: 退款金额
   - transaction_type: CANCEL 或 CANCEL_REFUND
   ↓
4. 执行 Lua 原子取消（atomicCancel）
   ↓
5. 返回新余额
```

**⚠️ 问题标注**:
- **未验证退款金额**: 可能退款金额 > 下注金额（数据异常）
- **缺少原订单状态检查**: 未验证原订单是否已结算

---

## 代码修改逻辑汇总

### 核心改造点

#### 1. **从数据库锁迁移到 Redis Lua 原子脚本**

**改造前（旧方案）**:
```php
// 数据库事务 + FOR UPDATE
DB::beginTransaction();
$wallet = PlayerPlatformCash::where('player_id', $playerId)->lockForUpdate()->first();
if ($wallet->money < $amount) {
    DB::rollBack();
    return error('余额不足');
}
$wallet->money -= $amount;
$wallet->save();
DB::commit();
```

**改造后（新方案）**:
```php
// Redis Lua 原子操作
$result = RedisLuaScripts::atomicBet($playerId, 'KT', [
    'order_no' => $orderNo,
    'amount' => $amount,
    'platform_id' => $platformId,
    'transaction_type' => TransactionType::BET,
]);
```

**改造优势**:
- ✅ 原子性保证（单个 Lua 脚本内执行）
- ✅ 性能提升（减少网络往返：3次 → 1次）
- ✅ 内置幂等性（订单锁机制）
- ✅ 自动同步队列（ZADD 到同步队列）

#### 2. **废弃 Service 层业务方法**

**改造前**:
```php
// 在 KTServiceInterface 中实现业务逻辑
public function bet($data) {
    // 余额检查
    // 扣款
    // 保存记录
}
```

**改造后**:
```php
// Service 方法标记为 @deprecated
public function bet($data): mixed {
    throw new \RuntimeException('bet() 方法已废弃，请使用 RedisLuaScripts::atomicBet');
}
```

**影响范围**:
- `bet()` - 已废弃
- `cancelBet()` - 已废弃
- `betResulet()` - 已废弃
- `reBetResulet()` - 已废弃

**⚠️ 问题标注**:
- **接口未同步**: `SingleWalletServiceInterface` 接口未更新，仍定义废弃方法
- **向后兼容性**: 直接抛出异常可能导致老代码崩溃

#### 3. **统一交易类型管理**

**改造前**:
```php
// 各平台自定义类型字段
$data['bet_type'] = 'prepay';
$data['settle_type'] = 'jackpot';
$data['cancel_type'] = 'refund';
```

**改造后**:
```php
// 统一使用 TransactionType 常量
use app\Constants\TransactionType;

$luaParams = [
    'transaction_type' => TransactionType::BET_PREPAY,  // 统一字段
];
```

**兼容处理**:
```php
// 使用 mapFromLegacy 转换旧字段
$transactionType = $data['transaction_type'] ?? TransactionType::mapFromLegacy($data);
```

#### 4. **添加参数验证和审计日志**

**新增辅助函数**:

```php
// 参数验证（app/functions.php）
validateLuaScriptParams($luaParams, [
```

**日志输出**:

## ⚠️ 潜在问题和隐患

### 🔴 严重问题

#### 1. **缓存一致性风险**

**问题描述**:
- Redis Lua 直接更新 `wallet:balance:{player_id}`
- 数据库更新（非 Lua 操作）后未同步更新缓存
- 可能导致余额不一致

**示例场景**:
```php
// 场景：管理员后台手动调整余额
PlayerPlatformCash::where('player_id', 123)->update(['money' => 1000]);
// ❌ 问题：Redis 缓存未更新，玩家仍看到旧余额
```

**影响范围**: 所有非 Lua 操作的余额变更（充值、提现、后台调整等）

---

#### 2. **订单号冲突（KT 平台）**

**问题描述**:
KT 平台 TakeWin=1 场景，立即结算时订单号处理不当

**代码位置**: `KTGameController.php:199`

```php
// Controller 手动追加 _settle
$settleOrderNo = $orderNo . '_settle';

// Lua 脚本也会追加 _settle（如果 bet 不存在）
if not string.find(finalOrderNo, '_settle$') then
    finalOrderNo = finalOrderNo .. '_settle'
end
```

**可能结果**: 订单号变成 `xxx_settle_settle`

---

#### 3. **diff 计算依赖 Redis（无降级）**

**问题描述**:
结算时 diff 计算依赖 Redis 中的 bet 记录

**代码位置**: `KTGameController.php:266-268`

```php
$betRecordKey = "game:record:bet:KT:{$orderNo}";
$betRecord = Redis::hGetAll($betRecordKey);
$betAmount = isset($betRecord['amount']) ? (float)$betRecord['amount'] : 0;
```

**风险场景**:
1. Redis 记录已过期（7天 TTL）
2. Redis 宕机或网络异常
3. 下注请求未成功但结算请求到达

**结果**: `betAmount = 0`，导致 `diff = win - 0 = win`（错误）

---

#### 4. **幂等锁时间固定（可能过短/过长）**

**问题描述**:
所有平台统一使用 300 秒（5 分钟）幂等锁

**代码位置**: `RedisLuaScripts.php:63`

```lua
redis.call('SETEX', KEYS[5], 300, 1)  -- 固定 300 秒
```

**潜在问题**:
- 快节奏游戏（如老虎机）：5分钟过长，可能误判重复
- 慢节奏游戏（如体育投注）：5分钟过短，真实重试被拒绝

---

#### 5. **退款金额未验证**

**问题描述**:
取消/退款时未验证退款金额是否合理

**代码位置**: `RedisLuaScripts.php:237`

```lua
local refundAmount = tonumber(ARGV[1])
-- ❌ 未检查 refundAmount 是否 > 原下注金额
local newBalance = currentBalance + refundAmount
```

**风险场景**: 恶意请求或数据异常，退款金额超过原下注金额

---

## 🔧 优化建议

### 优先级1（高优先级）

#### 1.1 **修复订单号冲突**

**方案**: 在 Controller 层判断是否需要追加后缀

## 平台差异对比

| 平台 | 控制器 | 特殊逻辑 | 注意事项 |
|------|--------|---------|---------|
| **KT** | `KTGameController` | TakeWin=1 立即结算 | 订单号冲突问题 |
| **DG** | `DGGameController` | 返回操作前余额 | 需缓存操作前余额 |

| **RSG** | `RsgGameController` | 打鱼机预扣款 | 使用 `BET_PREPAY` 类型 |
| **QT** | `QTGameController` | 免费游戏不扣款 | 使用 `BET_BONUS` 类型 |
| **MT** | `MtGameController` | 打赏扣款 | 使用 `BET_GIFT` 类型 |
| **BTG** | `BTGGameController` | 负数调整 | 使用 `BET_ADJUST` 类型 |
